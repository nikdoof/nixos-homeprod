# mx-01: Mail Server

mx-01 is a NixOS microVM (CID 12) on hyp-01, running on VLAN 106 with a publicly routed
IP address. It handles inbound and outbound email for two domains using Postfix (MTA) and
Dovecot (IMAP/LMTP), with spam/virus scanning, DKIM signing, DMARC policy enforcement,
and SPF validation all handled by Rspamd.

## Network

| Property     | Value                          |
|--------------|--------------------------------|
| Platform     | microVM, CID 12, VLAN 106      |
| IPv4         | 217.169.25.11/29               |
| IPv6         | 2001:8b0:bd9:106::3/64         |
| Gateway      | 217.169.25.9 (gw hosted VLAN)  |
| DNS          | 217.169.25.9, 2001:8b0:bd9:106::1 (Unbound on gw) |

Open TCP ports: **25** (SMTP/postscreen), **465** (SMTPS submission), **587** (submission),
**993** (IMAPS).

## Hosted domains

Virtual mailboxes are served for two domains:

- `doofnet.uk`
- `dimension.sh`

The host's own FQDN (`mx-01.doofnet.uk`) is the only entry in `mydestination`. All mail for
the virtual domains is delivered via LMTP to Dovecot.

## TLS / ACME

A single TLS certificate covers `mx-01.doofnet.uk`, obtained via ACME DNS-01 challenge
using the DigitalOcean API. After renewal, ACLs are applied so that `dovecot2` and
`postfix` can read the certificate files without running as `acme`, and all three services
(postfix, dovecot2, rspamd) are reloaded. The certificate is used by Postfix (inbound and
outbound), Dovecot (IMAP), and Rspamd (DKIM/DMARC verification).

## Postfix

### Listeners

| Port | Service     | Auth required | Notes                              |
|------|-------------|---------------|------------------------------------|
| 25   | postscreen  | No            | Pre-screens before reaching smtpd  |
| 587  | submission  | Yes (SASL)    | TLS enforced (`encrypt`)           |
| 465  | submissions | Yes (SASL)    | TLS wrapper mode                   |

Port 25 is handled by **postscreen** (not smtpd directly). The `smtp_inet` master.cf entry
is overridden to run `postscreen`, which passes clean connections to a `smtpd` pass service.
The `dnsblog` and `tlsproxy` helper processes are declared to support postscreen DNSBL
lookups.

### Relay policy

```
permit_mynetworks → permit_sasl_authenticated → reject_unauth_destination
```

`mynetworks` covers loopback, the private VLAN (`10.101.0.0/16`), and the hosted /29 and
/64, meaning infrastructure hosts can relay without authentication.

### Inbound TLS

- Security level: `may` (opportunistic; enforced for submission ports)
- Minimum protocol: TLSv1.2
- Session cache: `btree:/var/lib/postfix/data/smtpd_scache`
- `smtpd_tls_received_header = yes` — adds `Received:` trace with TLS info

### Outbound TLS (DANE)

Outbound delivery uses `smtp_tls_security_level = dane` with
`smtp_dns_support_level = dnssec`. For receiving servers that publish TLSA records, Postfix
performs DANE-verified TLS; for those that don't, it falls back to opportunistic TLS.
The DNSSEC-validating resolvers on mx-01 (Unbound on gw) are essential for this to work.

### Postscreen

Postscreen pre-screens inbound SMTP connections before they consume smtpd resources:

| Check                       | Action   | Detail                                                   |
|-----------------------------|----------|----------------------------------------------------------|
| DNSBL threshold ≥ 2         | enforce  | `zen.spamhaus.org*2`, `dnsbl.dronebl.org*2`, `bl.spamcop.net*1` |
| Blacklist action            | drop     | Silent drop for blacklisted IPs                          |
| `permit_mynetworks`         | pass     | Skips all postscreen checks for trusted networks         |

**Notes**: Bare newline checking is intentionally disabled at the postscreen level
(`postscreen_bare_newline_enable = no`). Keeping it enabled caused postscreen to defer
the BARLF check past STARTTLS negotiation, preventing TLS proxy from forwarding
connections to smtpd (resulting in `450 4.3.2` errors for legitimate senders like
O365). SMTP request smuggling protection is enforced by `smtpd_forbid_bare_newline = yes`
at the smtpd level instead, which is not subject to the same TLS proxy interaction.

Passed connections are cached in `btree:/var/lib/postfix/data/postscreen_cache`.

### HELO, sender, and recipient restrictions

**HELO restrictions** (in order):

1. `permit_mynetworks`
2. `reject_non_fqdn_helo_hostname`
3. `reject_invalid_helo_hostname`
4. `reject_rhsbl_helo dbl.spamhaus.org`
5. `permit`

**Sender restrictions** (in order):

1. `permit_mynetworks`, `permit_sasl_authenticated`
2. `reject_non_fqdn_sender`, `reject_unknown_sender_domain`
3. `reject_rhsbl_sender dbl.spamhaus.org`
4. `permit`

On port 25 (behind postscreen), the sender restrictions are overridden to be simpler
since SASL is not available and `reject_rhsbl_sender` can't be applied via the master.cf
`-o` syntax (no spaces allowed). Rspamd provides equivalent checks via its milter:

1. `permit_mynetworks`
2. `reject_non_fqdn_sender`, `reject_unknown_sender_domain`
3. `permit`

**Recipient restrictions** (in order):

1. `reject_invalid_hostname`, `reject_unknown_recipient_domain`
2. `reject_unauth_pipelining`, `reject_multi_recipient_bounce`
3. `permit_mynetworks`
4. `reject_non_fqdn_recipient`, `reject_unauth_destination`
5. RBL checks: `dnsbl.dronebl.org`, `zen.spamhaus.org`, `bl.spamcop.net`, `b.barracudacentral.org`, `dnsbl-1.uceprotect.net`
6. `permit`

SPF and additional RBL checks are handled by Rspamd rather than Postfix policy services.

### CVE-2023-51764 (SMTP request smuggling)

`smtpd_forbid_bare_newline = yes` is set globally. `mynetworks` hosts are exempted via
`smtpd_forbid_bare_newline_exclusions`.

### Rate limiting

Per-client limits measured over a 60-second window:

| Limit                 | Value |
|-----------------------|-------|
| Connection rate       | 10/min|
| Message rate          | 100/min|
| Recipient rate        | 100/min|
| Auth attempt rate     | 10/min|

### Bounce / error handling

| Setting                    | Value | Rationale                            |
|----------------------------|-------|--------------------------------------|
| `bounce_queue_lifetime`    | 1d    | Short bounce window reduces backscatter risk |
| `smtpd_soft_error_limit`   | 3     | Delay after 3 soft errors            |
| `smtpd_hard_error_limit`   | 10    | Drop after 10 hard errors            |
| `smtpd_error_sleep_time`   | 1s    | Slow-down delay per error            |

### Milters

A single Rspamd proxy milter is applied to both `smtpd_milters` and `non_smtpd_milters`:

```
inet:127.0.0.1:11332
```

Rspamd handles all mail filtering functions previously split across three separate milters:

| Function     | Previously         | Now                              |
|--------------|--------------------|----------------------------------|
| DKIM signing | OpenDKIM           | Rspamd `dkim_signing` module     |
| DKIM verify  | OpenDKIM           | Rspamd built-in                  |
| DMARC        | OpenDMARC          | Rspamd `dmarc` module            |
| SPF          | policyd-spf        | Rspamd built-in                  |
| Spam scoring | —                  | Rspamd (bayes, neural, rules)    |
| Greylisting  | —                  | Rspamd `greylist` module         |
| Rate limit   | —                  | Rspamd `ratelimit` module        |

`milter_macro_daemon_name = ORIGINATING` is set on submission ports so Rspamd can
distinguish outbound authenticated mail from inbound.

`milter_default_action = accept` means a milter failure does not block delivery.

### Aliases

Local system aliases:

| Alias     | Destination                       |
|-----------|-----------------------------------|
| `root`    | `root-mail@m.tensixtyone.com`     |
| `inbox`   | `paperless`, `household@williams.id` |
| `nikdoof` | `andy@williams.id`                |
| `salkunh` | `jo@williams.id`                  |

Virtual aliases forward `root@` for internal subdomains to `root-mail@m.tensixtyone.com`.

## Rspamd

Rspamd replaces OpenDKIM, OpenDMARC, and policyd-spf, consolidating all mail filtering
into a single milter-backed service. It uses Valkey as its backing store and exposes a
controller UI on `127.0.0.1:11334`.

### Architecture

Two worker processes run:

| Worker   | Socket           | Purpose                                        |
|----------|------------------|------------------------------------------------|
| Proxy    | `127.0.0.1:11332`| Milter interface for Postfix; self-scans       |
| Controller | `127.0.0.1:11334`| Web UI (admin restricted to localhost)         |

The proxy worker is configured as a self-scan upstream — it receives messages from Postfix,
scans them locally, and returns the verdict. No separate scanning nodes.

### DKIM signing

Rspamd replaces OpenDKIM for DKIM signing. Configuration is in `dkim_signing.conf`:

| Setting   | Value                                              |
|-----------|----------------------------------------------------|
| Selector  | `658553b1d7df5d1bc6bdb2176d87b585a0ad54ce` (SHA1 of `mx-01.doofnet.uk`) |
| Key path  | `/persist/opendkim/keys/<selector>.private`        |
| Domains   | `doofnet.uk`, `dimension.sh` (all virtual domains) |
| Condition | Sign all outbound mail (`return true`)             |

The key files are the same ones originally created for OpenDKIM, stored on `/persist` for
survival across rebuilds. The DKIM public key must be published as a DNS TXT record at
`<selector>._domainkey.<domain>` for each virtual domain.

### DMARC

Rspamd's `dmarc` module is configured with reporting enabled and rejection actions:

| Setting                              | Value               |
|--------------------------------------|---------------------|
| Reporting enabled                    | `true`              |
| Reject action for DMARC fail         | `reject:reject`     |

The DMARC policy is **monitoring mode** (`p=none`) at the DNS level. Rspamd logs DMARC
failures but applies `reject` only within its own scoring system — the actual DNS `p=`
policy controls what receiving servers do.

### SPF

SPF checking is built into Rspamd's core scanning — no separate policy daemon needed.
Rspamd evaluates SPF for each inbound message and factors the result into the spam score.

### Greylisting

Greylisting is enabled via the `greylist` module (config in `greylist.conf`). Note the
config file is named `greylist.conf`, not `greylisting.conf` — the upstream module expects
the shorter name.

### Neural network

Rspamd's neural network classifier is configured with Valkey as its backend:

```
servers = "127.0.0.1:6379";
```

This enables per-user neural classification models persisted in Valkey.

### MX check

`mx_check` is enabled: Rspamd verifies the sender domain has reachable MX records. Mail
from domains without valid MX configuration is scored accordingly.

### Per-user outbound rate limiting

Rspamd enforces per-user rate limits on authenticated outbound mail via `ratelimit.conf`:

| Setting          | Value  |
|------------------|--------|
| Burst            | 200    |
| Rate             | 100/hr |

This prevents compromised accounts from flooding outbound queues.

### Settings for authenticated users

Rspamd's `settings.conf` defines a high-priority rule for authenticated users that:

- Disables RBL, SPF, and neural checks (trusted outbound)
- Raises reject threshold to 100 (practically disables rejection)
- Raises greylist threshold to 50

This ensures authenticated outbound mail is not delayed or blocked by false positives.

### Redis/Valkey

Rspamd uses Valkey (a Redis-compatible key-value store) for:

- Greylisting state
- Neural network classification models
- Rate limiting counters
- Statistics and learning data

Configuration is in `redis.conf`:

```
servers = "127.0.0.1:6379";
```

Valkey runs as a `redis.servers.rspamd` systemd unit with AOF + RDB persistence, writing
to `/persist/valkey` via a bind mount at `/var/lib/redis-rspamd`.

### Milter headers

Rspamd adds the following headers to processed messages (config in `milter_headers.conf`):

- `X-Spam-Score` — numeric spam score
- `X-Spam-Level` — visual star representation of score
- `Authentication-Results` — SPF/DKIM/DMARC auth status

## Dovecot

Dovecot provides IMAP access and LMTP delivery for Postfix. POP3 is disabled. PAM
authentication is disabled.

### Virtual users

All mailboxes are virtual users authenticated against an `passwd-file` stored in an
age-encrypted secret (`secrets/mx01DovecotPasswd.age`). Usernames are normalised to
lowercase full email addresses (`auth_username_format = %Lu`).

Mail is stored under:
```
/persist/vmail/<domain>/<user>/Maildir/
```

The `vmail` system user owns all mailbox files. On service start, an `ExecStartPre` step
`chown`s the entire mail store to `vmail:vmail`.

### Mailboxes

The following IMAP special-use folders are auto-created for all users:

| Folder    | RFC role   | Auto-expunge |
|-----------|------------|--------------|
| All       | `\All`     | Never        |
| Archive   | `\Archive` | Never        |
| Drafts    | `\Drafts`  | Never        |
| Flagged   | `\Flagged` | Never        |
| Junk      | `\Junk`    | 60 days      |
| Sent      | `\Sent`    | Never        |
| Trash     | `\Trash`   | 60 days      |

### LMTP socket

Dovecot exposes an LMTP listener at `/var/spool/postfix/dovecot-lmtp` (inside the Postfix
chroot), owned `postfix:postfix`. Postfix delivers via `lmtp:unix:/var/spool/postfix/dovecot-lmtp`.

### Authentication socket

The SASL auth socket is at `/var/spool/postfix/auth`, owned `postfix:postfix`. Postfix uses
this for both submission port SASL and inbound smtpd SASL.

### TLS

Minimum TLS version: TLSv1.2. Server cipher preference is enabled.

### Plugins / features

**Full-text search (FTS)** via `fts_flatcurve` provides indexed IMAP SEARCH across all
mailboxes. Two tokenizers are configured:

| Tokenizer      | Algorithm          | Max length |
|----------------|--------------------|------------|
| `generic`      | `simple`           | 30 chars   |
| `email-address`| (default)          | 100 chars  |

Token filters for indexing and search are language-specific:

| Language | Filters                                              |
|----------|------------------------------------------------------|
| English  | `lowercase`, `snowball` (stemming), `english-possessive`, `stopwords` |
| German   | `normalizer-icu`, `snowball`, `stopwords`            |

Supported languages: English (`en`), German (`de`).

**Listescape** is enabled in IMAP to allow mailbox names containing characters that would
normally be interpreted as hierarchy separators (e.g., `foo.bar` or `foo/bar`). The plugin
allows escaping these characters so they can be used literally in mailbox names.

**IMAP ACL** (Access Control List) enables shared mailbox access. Users can subscribe to
shared mailboxes under the `Shared/` namespace using the `/` hierarchy separator.

**Shared mailboxes** are configured via `doofnet.mailserver.sharedAccess`. Currently
`nikdoof@doofnet.uk` has access to:
- `paperless@doofnet.uk`
- `dmarc-reports@doofnet.uk`
- `hello@nikdoof.com`

The shared namespace uses a dedicated INDEX directory to avoid locking conflicts.

### Sieve spam filtering

A global `sieve_before` script (`spam-to-junk.sieve`) is applied to all LMTP deliveries:

```
if header :contains "X-Spam-Level" "*****" {
  fileinto :create "Junk";
  stop;
}
```

Mail with an `X-Spam-Level` of 5+ stars (spam score ≥ ~5) is automatically filed into
the Junk folder. The Sieve plugin is loaded per-protocol for LMTP only (the pigeonhole
plugin is compiled for lda/lmtp and cannot be loaded in IMAP contexts).

Per-user Sieve scripts are supported but not currently configured — users can place custom
scripts at `~/.dovecot.sieve` or in the `~/sieve/` directory.

### Metrics endpoint

Dovecot exposes an HTTP stats endpoint on port 9166 (`service stats`). The following
metrics are collected:

| Metric          | Filter                              | Grouped by                                |
|-----------------|-------------------------------------|-------------------------------------------|
| `auth_success`  | auth_request_finished, success=yes  | —                                         |
| `imap_command`  | imap_command_finished               | cmd_name, tagged_reply_state              |
| `smtp_command`  | smtp_server_command_finished        | cmd_name, status_code, duration histogram |
| `mail_delivery` | mail_delivery_finished              | duration histogram                        |

## Fail2ban

The `dovecot` jail is enabled via `doofnet.fail2ban.jails.dovecot = true`, banning IPs that
exceed failed IMAP authentication thresholds.

## Persistence

mx-01 uses the standard microVM `/persist` virtiofs share for state that must survive
rebuilds:

| Path                   | Contents                        |
|------------------------|---------------------------------|
| `/persist/vmail/`      | All user mailboxes (Maildir)    |
| `/persist/opendkim/keys/` | DKIM private keys            |
| `/persist/valkey/`     | Valkey AOF + RDB dumps          |
| `/persist/ssh_host_*`  | SSH host keys (standard microVM)|
| `/persist/acme/`       | ACME certificates               |

## Observability

Three exporters run locally and are scraped by Alloy:

| Exporter                      | Port | Job       | Metrics                                    |
|-------------------------------|------|-----------|--------------------------------------------|
| `prometheus-postfix-exporter` | 9154 | `postfix` | Queue lengths, delivery rates, error counts (systemd journal) |
| Dovecot stats HTTP            | 9166 | `dovecot` | Auth, IMAP command, SMTP command, delivery metrics |
| `prometheus-dmarc-exporter`   | auto | `dmarc`   | DMARC report pass/fail counts (reads IMAP mailbox `dmarc-reports@doofnet.uk`) |

The DMARC exporter connects to Dovecot via IMAP using credentials loaded as a systemd
credential file (compatible with `DynamicUser`), reading aggregate DMARC reports delivered
to the `dmarc-reports@doofnet.uk` mailbox.

## DNS records required

The following DNS records must be published for correct mail operation:

| Zone          | Name                            | Type  | Value / Notes                          |
|---------------|---------------------------------|-------|----------------------------------------|
| `doofnet.uk`  | `@`                             | MX    | `mx-01.doofnet.uk`                     |
| `dimension.sh`| `@`                             | MX    | `mx-01.doofnet.uk`                     |
| `doofnet.uk`  | `@`                             | TXT   | SPF record (`v=spf1 mx ~all` or similar)|
| `dimension.sh`| `@`                             | TXT   | SPF record                             |
| `doofnet.uk`  | `<selector>._domainkey`         | TXT   | DKIM public key (selector = SHA1 of `mx-01.doofnet.uk`) |
| `dimension.sh`| `<selector>._domainkey`         | TXT   | DKIM public key                        |
| `doofnet.uk`  | `_dmarc`                        | TXT   | `v=DMARC1; p=none; rua=mailto:dmarc-reports@doofnet.uk` (monitoring mode) |
| `dimension.sh`| `_dmarc`                        | TXT   | DMARC policy                           |
| Reverse       | `11` in `8-15.25.169.217.in-addr.arpa` | PTR | `mx-01.doofnet.uk.`             |

The DKIM selector is derived at build time as the SHA1 hash of the string `mx-01.doofnet.uk`
and is therefore stable as long as the hostname does not change.
